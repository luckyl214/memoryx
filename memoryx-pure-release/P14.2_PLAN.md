# P14.2 Feishu Real Runtime Bridge

## 目标
完成飞书 ↔ Hermes ↔ MemoryX 全链路真实运行时集成。

## P1-1: 附件真实下载落盘

### 当前状态
- `AttachmentRef` 有 `image_key` / `file_key` / `name` / `size`
- 飞书队列双写到 `feishu_attachments` 表
- 但 Hermes 看到的是 key 而不是本地文件内容

### 需要实现
```
message_id + file_key/image_key
  → 调用飞书下载资源 API
  → 保存到 spool_dir（带 sha256 哈希）
  → feishu_attachments.status = "downloaded"
  → 返回 local_path 给 Hermes runner
```

### 飞书 API
- 下载机器人上传的图片：`GET /open-apis/im/v1/images/{image_key}`
- 获取消息中的资源文件：需要先用消息接口获取 file_key，再下载

### 实现步骤
1. 在 `FeishuClient` 添加 `download_resource(image_key/file_key)` 方法
2. 在 `FeishuSQLiteQueue` 添加附件下载状态跟踪
3. 在 `FeishuRenderJob` 添加 `local_path` 字段
4. 在 `FeishuHermesBotService` 处理附件下载逻辑

## P1-2: Hermes runner 真接入

### 当前状态
- P14 smoke 测试用的是 mock runner
- 没有真实 Hermes stream/tool/guard 集成

### 需要实现
```
on_user_message       → context_block
on_tool_call          → decision / guard_block
on_assistant_response → claim guard
on_session_end        → narrative reflection
```

### 实现步骤
1. 创建 `HermesRunner` 真实实现（替代 smoke runner）
2. 集成 P12.1 MemoryX bridge 事件
3. 实时映射到飞书卡片更新
4. 添加 stream 清洗和工具调用记录

## 验收标准

| 场景 | 必须结果 |
|------|----------|
| 飞书发文本 | queued 卡片 → running → done，结构化正文 |
| 飞书发图片 | 图片下载到 local_path，Hermes 能读取 |
| 飞书发文件 | 文件下载到 local_path，Hermes 能读取 |
| Hermes 调工具 | 卡片显示 tool_start/tool_end/耗时/摘要 |
| MemoryX guard | 卡片显示 claim_guard / lesson_enforcement 决策 |
| 飞书重试事件 | 不重复生成卡片 |
| Hermes 报错 | 卡片变 error，附件保留，job 可重试/DLQ |
| stream 泄漏 | `<|analysis|>`、tool JSON、scratchpad 不出现 |
